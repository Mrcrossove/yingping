# 小程序未完成项技术实现规划

## 目标

把当前小程序端明显半实现的页面补成可上线使用的业务闭环，优先保证用户下单、推广员收益、商户资料、订单处理这些核心流程真实落库、可追踪、可验收。

本规划覆盖以下页面和后端能力：

- `miniapp/src/pages/promotion/promotion.vue`
- `miniapp/src/pages/cart/cart.vue`
- `miniapp/src/pages/merchant-apply/merchant-apply.vue`
- `miniapp/src/pages/order-detail/order-detail.vue`
- `miniapp/src/pages/employee/employee.vue`
- `server` 中订单、推广、商户资料、用户选择、支付相关接口

## 当前约束

1. 微信支付参数、微信小程序合法域名、商户 API 证书暂未完整配置，所以支付不能作为第一阶段闭环目标。
2. `Order` 目前只有 `note`，没有收货地址快照字段；如果购物车要选择地址，必须让订单保存地址信息，否则后续配送员和订单详情看不到用户选择的地址。
3. `User` 目前没有店铺地址、店铺简介等商户资料字段；如果商户申请页要完整保存资料，需要新增商户资料表或字段。
4. 推广员上传商家当前后端只原样返回，没有落库；如果要让老板或管理员后续审核跟进，需要新增“推广商家线索”数据表和后台/接口管理能力。
5. 目前已有角色权限修复基础，后续新增接口仍要按角色收口，不能让普通商户调用业务员、管理员接口。

## 推荐实施顺序

### 阶段 1：补订单地址和备注闭环

优先级：最高。

原因：购物车下单是小程序最核心流程。地址和备注直接影响商户下单、制作配送、订单详情展示，也会影响后续支付和配送操作。

后端改动：

- 修改 `server/prisma/schema.prisma` 的 `Order` 模型，新增地址快照字段：
  - `addressId Int?`
  - `receiverName String?`
  - `receiverPhone String?`
  - `receiverAddress String?`
- 在订单创建接口 `POST /api/orders` 中接收：
  - `addressId`
  - `note`
  - `items`
- 在 `OrderService.create()` 中：
  - 校验 `addressId` 属于当前下单用户。
  - 读取地址并写入订单快照。
  - 地址删除或修改后，不影响历史订单展示。
- 在订单列表和详情返回中包含收货人、电话、地址、备注。

小程序改动：

- `pages/cart/cart.vue`
  - 增加地址选择区。
  - 默认加载用户默认地址。
  - 没有地址时提示先添加地址。
  - 增加订单备注输入框。
  - 下单时提交 `addressId` 和 `note`。
- `pages/order-detail/order-detail.vue`
  - 展示收货人、手机号、完整地址、备注。
- `pages/employee/employee.vue`
  - 配送员/业务员工作台订单卡片展示配送地址。

验收标准：

- 有默认地址时，购物车进入页面能自动带出默认地址。
- 没有地址时，不能直接下单，并提示先新增地址。
- 下单成功后订单详情能看到当时选择的地址和备注。
- 修改或删除地址后，历史订单地址不变化。

### 阶段 2：补推广员页面真实数据和上传商家落库

优先级：高。

原因：这是当前最明显的半实现页面。推广员收益和商家线索如果不落库，业务无法运营。

后端改动：

- 新增 Prisma 模型 `PromoterMerchantLead`，建议字段：
  - `id`
  - `promoterId`
  - `merchantName`
  - `contactName`
  - `phone`
  - `address`
  - `remark`
  - `status`：`pending`、`followed`、`converted`、`rejected`
  - `createdAt`
  - `updatedAt`
- 新增接口：
  - `POST /api/promotion/upload-merchant`
  - `GET /api/promotion/my-merchant-leads`
  - `GET /api/promotion/commission-details`
- 修改 `PromotionService.uploadMerchant()`：
  - 校验当前用户必须是 `promoter`、`boss` 或 `admin` 中允许提交的角色，建议先限制为 `promoter`。
  - 校验商家名称和手机号必填。
  - 手机号重复时提示已提交，避免重复线索。
  - 真实写入 `PromoterMerchantLead`。
- 后续可在后台新增“推广商家线索管理”，本阶段先保证小程序端提交和列表可查。

小程序改动：

- `pages/promotion/promotion.vue`
  - 分润记录移除写死 `commissionRecords`。
  - 页面加载时调用 `promotionApi.commissionDetails()`。
  - 上传商家调用真实 `promotionApi.uploadMerchant()`。
  - 上传成功后清空表单并刷新“我的上传商家”列表。
  - 增加空状态和加载状态。
- `miniapp/src/api/index.ts`
  - 增加 `uploadMerchant` 和 `myMerchantLeads` API 方法。

验收标准：

- 推广员进入页面能看到真实推广码。
- 分润记录从后端读取，不再显示固定示例数据。
- 上传商家后刷新页面仍能看到该线索。
- 后端数据库能查到对应推广员提交的商家线索。

### 阶段 3：补商户申请资料存储

优先级：高。

原因：商户申请如果只保存账号、名称、手机号，会导致后台审核信息不足，也无法给业务员/老板判断商户真实情况。

推荐方案：新增独立商户资料表，不直接把所有店铺字段塞进 `users` 表。

后端改动：

- 新增 Prisma 模型 `MerchantProfile`，建议字段：
  - `id`
  - `userId`
  - `shopName`
  - `shopAddress`
  - `description`
  - `licenseImage`
  - `contactName`
  - `contactPhone`
  - `createdAt`
  - `updatedAt`
- 修改注册接口：
  - `POST /api/auth/register` 接收 `shopAddress`、`description`、可选 `licenseImage`。
  - 创建商户用户后同步创建 `MerchantProfile`。
- 修改待审核商户列表：
  - `GET /api/users/pending-merchants` 返回 `merchantProfile`。
- 修改审核详情：
  - 管理后台能查看店铺地址、简介和证照图。

小程序改动：

- `pages/merchant-apply/merchant-apply.vue`
  - 提交 `shopAddress` 和 `description`。
  - 如果营业执照上传已接入文件接口，可增加 `licenseImage` 上传。
  - 提交成功后提示等待审核。

后台改动：

- `admin` 商户审核页面展示：
  - 店铺地址
  - 店铺简介
  - 营业执照图片

验收标准：

- 商户申请时填写的店铺地址和简介能进入数据库。
- 管理后台审核商户时能看到这些资料。
- 审核通过后商户可以正常登录和下单。

### 阶段 4：补订单详情角色动作

优先级：中高。

原因：订单详情页是各角色处理订单的主要入口。列表页有操作不够，详情页必须能完成角色自己的核心动作。

后端改动：

- 复用现有订单接口，不重复新增无意义接口：
  - 业务员/老板/管理员：接单、派制作、派配送。
  - 制作员：开始制作、制作完成。
  - 配送员：开始配送、配送完成。
- 需要新增的接口：
  - `POST /api/orders/:id/cancel`：商户取消未接单/未支付订单。
  - `POST /api/orders/:id/reorder` 可不作为后端接口，前端读取原订单商品重新加入购物车更简单。
- 如果支付资料齐全后，再接：
  - `POST /api/payments/create/:orderId`
  - `POST /api/payments/jsapi/:orderId`

小程序改动：

- `pages/order-detail/order-detail.vue`
  - 根据当前用户角色和订单状态显示不同按钮。
  - 商户：
    - 待支付/待接单：取消订单。
    - 已创建待支付：去支付。
    - 任意已完成订单：再次下单。
  - 业务员：
    - 待接单：接单。
    - 已接单：选择制作员/配送员并派单。
  - 制作员：
    - 已派单：开始制作。
    - 制作中：制作完成。
  - 配送员：
    - 待配送：开始配送。
    - 配送中：配送完成。
- 所有动作成功后刷新详情。

验收标准：

- 每个角色只能看到自己能执行的按钮。
- 点击无权限接口不会出现误操作。
- 订单状态变更后页面立即刷新为最新状态。

### 阶段 5：优化员工页手动录单

优先级：中。

原因：手动输入商户 ID、商品 ID 对普通用户不可用，容易录错单。这个功能应做成选择器，降低业务员录单成本。

后端改动：

- 新增或复用查询接口：
  - `GET /api/users/merchants`：业务员、管理员、老板可查询可下单商户。
  - `GET /api/products`：已有商品列表接口可复用。
- 修改 `POST /api/orders/manual`：
  - 校验当前用户角色必须为 `salesperson`、`admin`、`boss`。
  - 校验 `merchantId` 对应用户角色为 `merchant` 且状态正常。
  - 支持多个商品项，而不是单个商品 ID。
  - 支持 `addressId` 或手动地址快照，便于代客录单。

小程序改动：

- `pages/employee/employee.vue`
  - 商户 ID 输入改成商户选择器。
  - 商品 ID 输入改成商品选择器/商品列表多选。
  - 支持数量加减。
  - 显示订单金额合计。
  - 支持备注。
  - 如果代客录单需要配送地址：
    - 方案 A：选择该商户已有地址。
    - 方案 B：业务员手动填写收货人、电话、地址。
  - 推荐先做方案 B，避免跨用户地址权限复杂。

验收标准：

- 业务员不用输入任何 ID 也能完成代客录单。
- 多商品、多数量金额计算正确。
- 后端保存的订单 merchantId、salespersonId、商品明细正确。

### 阶段 6：支付入口预留与正式支付闭环

优先级：中，依赖微信支付资料。

原因：支付功能需要微信商户号、API v3 密钥、商户 API 证书、回调域名、小程序合法域名配置完整后才能真实验证。

当前先做：

- 购物车和订单详情保留“去支付”入口。
- 如果后端返回“微信支付参数未配置完整”，前端提示“支付暂未开通，请联系商家处理”，不要让页面崩溃。

资料齐全后再做：

- 实现标准微信支付 v3 回调：
  - 校验微信平台证书序列号。
  - 校验回调签名。
  - 使用 API v3 key 解密 resource。
  - 根据 `out_trade_no` 更新支付单和订单状态。
- 小程序端调用 `wx.requestPayment`。
- 支付成功后刷新订单状态。

验收标准：

- 未配置微信支付时，按钮有明确提示，不影响下单。
- 配置完整后，小程序能发起 JSAPI 支付。
- 微信支付成功回调能更新订单和支付记录。

## 权限规则

- 商户：
  - 只能查看和操作自己的订单。
  - 只能使用自己的地址下单。
  - 可以取消未被接单或未支付的订单。
- 推广员：
  - 只能查看自己的推广码、分润记录和上传商家线索。
  - 不能查看其他推广员线索。
- 业务员：
  - 可以接单、派单、代客录单。
  - 只能查看自己接手或创建的订单。
- 制作员：
  - 只能处理派给自己的制作订单。
- 配送员：
  - 只能处理派给自己的配送订单。
- 老板/管理员：
  - 可以查看和管理全部订单、商户、推广线索。

## 前端体验要求

1. 所有真实接口页面都要有加载状态、空状态、失败重试或失败提示。
2. 不再展示固定示例数据，除非明确作为接口失败兜底，并且不能误导用户认为是真实数据。
3. 选择器优先于手动输入 ID。
4. 提交按钮要防重复点击。
5. 角色按钮和开发调试入口不进入正式版本。
6. 支付未配置时只提示功能未开通，不影响用户提交订单。

## 推荐执行节奏

1. 先做阶段 1：订单地址、备注、订单详情展示。
2. 再做阶段 2：推广员真实分润记录和上传商家落库。
3. 再做阶段 3：商户申请资料保存和后台审核展示。
4. 再做阶段 4：订单详情角色动作补齐。
5. 再做阶段 5：员工页手动录单选择器。
6. 最后等微信支付资料齐全后做阶段 6：真实支付闭环。

这个顺序可以先把不依赖外部平台参数的业务能力做完，支付能力等域名和商户证书齐全后再接入，避免现在把支付写成半成品。
