# P0 订单取消退款一致性修复规划执行文件

## 背景

当前系统存在订单状态与支付状态脱节风险：订单支付成功后 `payment.status = paid`，但订单仍可能处于 `pending`。如果商户此时取消订单，库存会回滚，但资金不会自动退款，形成“订单取消、钱未退”的 P0 财务一致性问题。

## 修复目标

1. 未支付订单允许取消，取消后订单进入 `cancelled`。
2. 已支付订单不允许走普通取消接口，必须走退款接口。
3. 已支付未接单订单允许商户在小程序发起退款。
4. 退款发起后订单进入 `cancelled`，支付进入 `refunding`。
5. 微信退款成功回调后支付进入 `refunded`，订单保持 `cancelled`。
6. 订单详情和订单列表展示支付状态，让用户知道“待支付 / 已支付 / 退款中 / 已退款”。
7. 后台退款也同步订单状态，避免后台和小程序行为不一致。

## 实施顺序

### 阶段 1：后端取消接口加支付保护

- `OrderService.cancel()` 查询订单时带上 `payment`。
- 如果存在 `payment.status = paid/refunding/refunded`，普通取消接口拒绝处理。
- 未支付订单仍允许取消。

### 阶段 2：退款服务同步订单状态

- `PaymentService.refund()` 查询 payment 时包含 order。
- 只允许 `paid` 支付退款。
- 商户只能退款自己的订单，老板/管理员可退款全部订单。
- 商户自助退款只允许订单仍处于 `pending`，已接单后需要后台处理。
- 微信退款请求成功后：
  - payment 改为 `refunding`
  - order 改为 `cancelled`
  - 回滚库存仅在 `stockDeducted = true` 时执行
  - 写入订单流转记录

### 阶段 3：增加退款回调入口

- 新增 `/payments/refund-notify`。
- 复用微信支付回调验签和 AES-GCM 解密。
- `refund_status = SUCCESS` 时将 payment 改为 `refunded`。
- 失败或处理中状态先不误标成功。

### 阶段 4：小程序支付状态展示和操作入口

- 订单列表带上支付状态显示。
- 订单详情展示支付状态。
- 订单列表补“取消订单 / 申请退款”按钮。
- 订单详情：
  - 未支付 pending：显示“取消订单”和“去支付”
  - 已支付 pending：显示“申请退款”
  - refunding：显示“退款中”
  - refunded：显示“已退款”

### 阶段 5：后台提示文案修正

- 后台支付记录退款文案改为真实微信原路退款，不再提示“只更新系统状态”。

## 验收清单

- 未支付订单可以取消，库存正确回滚。
- 已支付订单调用普通取消接口返回错误。
- 已支付未接单订单可在小程序申请退款。
- 退款发起成功后订单为 `cancelled`，支付为 `refunding`。
- 退款回调成功后支付为 `refunded`。
- 订单列表和详情能看到支付状态。
- 后端、后台、小程序构建通过。
